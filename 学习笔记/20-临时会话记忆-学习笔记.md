# LangChain 临时会话记忆（Session Memory）学习笔记

> 对应代码：`20-临时会话记忆.py`
> 主题：使用 `RunnableWithMessageHistory` 实现多轮对话的"记忆"功能（内存存储，进程结束即丢失）

---

## 1. 核心概念

这段代码解决的问题是：**大模型本身是无状态的**，每次调用 `invoke` 都是独立请求，不会自动记得上一轮说了什么。要实现"多轮对话"，需要手动把历史消息拼回 prompt 里。

LangChain 提供了 `RunnableWithMessageHistory` 来自动完成这件事：每次调用前，自动从存储里取出历史消息塞进 prompt；调用后，自动把这一轮的问答存回去。

| 概念 | 说明 |
|---|---|
| `session_id` | 区分不同会话/用户的标识，相同 `session_id` 共享同一段历史 |
| `InMemoryChatMessageHistory` | 历史消息的载体，存在内存里（程序重启即清空） |
| `store` 字典 | 全局容器，`{session_id: InMemoryChatMessageHistory}` |
| `MessagesPlaceholder` | prompt 模板中的占位符，运行时会被历史消息列表填充 |
| `RunnableWithMessageHistory` | 包装链路，自动管理"取历史 → 调用 → 存历史"流程 |

---

## 2. 整体流程图

```
用户输入 input
      │
      ▼
RunnableWithMessageHistory.invoke({"input": ...}, session_config)
      │
      ├─① 根据 session_id 调用 get_history() 拿到该会话的历史记录
      │
      ├─② 把历史记录注入 chat_history 占位符，把 input 注入 prompt
      │
      ▼
chain = prompt | print_prompt | model | strParser
      │         （拼装完整提示词） （打印调试） （调用千问模型） （解析为字符串）
      ▼
   返回结果 res
      │
      └─③ 自动把本轮 (human: input, ai: res) 存回 store[session_id]
```

---

## 3. 代码分段解析

### 3.1 初始化模型

```python
from langchain_community.chat_models.tongyi import ChatTongyi

model = ChatTongyi(
    model='qwen-plus'
)
```
使用阿里通义千问的 `qwen-plus` 模型作为对话模型。

### 3.2 构建 Prompt 模板

```python
prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你需要根据会话历史回应用户问题。对话历史："),
        MessagesPlaceholder("chat_history"),
        ("human", "请回答如下问题：{input}")
    ]
)
```

- `system` 消息：设定角色/任务背景
- `MessagesPlaceholder("chat_history")`：**关键**，运行时会被替换成历史消息列表（多条 human/ai 消息）
- `human` 消息：本轮用户的实际问题，用 `{input}` 占位

> 💡 注释里被注掉的 `PromptTemplate.from_template(...)` 是另一种写法（纯字符串拼接历史），但不如 `MessagesPlaceholder` 规范，因为后者能保留消息的角色信息（谁说的），而不是把历史压成一段文本。

### 3.3 构建链路（Chain）

```python
def print_prompt(full_prompt):
    print("="*20, full_prompt.to_string(), "="*20)
    return full_prompt

chain = prompt | print_prompt | model | strParser
```

这是 LCEL（LangChain Expression Language）的管道写法，`|` 表示"前一步的输出作为后一步的输入"：

1. `prompt` → 生成完整的提示词对象
2. `print_prompt` → 一个调试用的 `RunnableLambda`，打印出当前完整 prompt，方便观察历史是否被正确拼接，然后原样传递下去
3. `model` → 调用通义千问模型
4. `strParser`（`StrOutputParser`）→ 把模型返回的消息对象解析成纯字符串

### 3.4 历史存储管理

```python
store = {}  # key是session_id，value是InMemoryChatMessageHistory对象

def get_history(session_id):
    if session_id not in store:
        store[session_id] = InMemoryChatMessageHistory()
    return store[session_id]
```

简单的"懒加载"模式：第一次访问某个 `session_id` 时新建一个历史对象，之后复用。

⚠️ **注意**：`store` 是一个普通 Python 字典，存在进程内存中。这意味着：
- 程序重启后历史全部丢失（这也是标题"**临时**会话记忆"的含义）
- 不适合生产环境多进程/多机部署（每个进程的 `store` 互不相通）
- 生产环境通常会换成 Redis、数据库等持久化存储

### 3.5 包装出带记忆的链路

```python
conversation_chain = RunnableWithMessageHistory(
    chain,
    get_history,
    input_messages_key="input",
    history_messages_key="chat_history"
)
```

| 参数 | 作用 |
|---|---|
| `chain` | 被包装的原始链路 |
| `get_history` | 获取历史记录的函数（输入 session_id，输出 History 对象） |
| `input_messages_key="input"` | 告诉它 `invoke()` 传入字典里哪个 key 是"本轮用户输入" |
| `history_messages_key="chat_history"` | 告诉它历史记录要填到 prompt 模板里的哪个占位符 |

### 3.6 调用示例

```python
session_config = {
    "configurable": {
        "session_id": "user_001"
    }
}

res = conversation_chain.invoke({"input": "小明有3只猪"}, session_config)
res = conversation_chain.invoke({"input": "小明有5只猪"}, session_config)
res = conversation_chain.invoke({"input": "总共有几只猪"}, session_config)
```

> `session_config` 的写法是固定格式：必须套一层 `"configurable": {"session_id": "..."}`，这是 LangChain `Runnable` 配置参数的标准写法。

**执行过程中，三次调用的历史会逐步累积：**

| 调用轮次 | 用户输入 | 此时 chat_history 包含的内容 | 模型能"看到"的信息 |
|---|---|---|---|
| 第1次 | 小明有3只猪 | （空） | 只有这句话本身 |
| 第2次 | 小明有5只猪 | [human: 小明有3只猪, ai: 第1次回复] | 知道前面提到过3只猪 |
| 第3次 | 总共有几只猪 | 上面两轮的 4 条消息 | 能结合前两轮推断出"猪的数量"语境 |

> 这里有意思的一点是：第3次问"总共有几只猪"，结果会**依赖模型如何理解上下文**——是把"3只"和"5只"相加（8只），还是认为"5只"是更新后的数量（即更正了第一次的说法）。这正是 `MessagesPlaceholder` + 历史记忆要解决的"歧义消解"问题，也是这个示例想演示的效果。建议实际运行代码观察模型的回答来理解。

## 4. 知识点小结（速记）

- ✅ LLM 本身无状态 → 多轮对话靠"手动拼接历史"实现
- ✅ `RunnableWithMessageHistory` = 自动化的"取历史 + 存历史"包装器
- ✅ 历史存储的核心是 `get_history(session_id)` 这个函数，可以自由替换底层存储方式（内存 / Redis / 数据库）
- ✅ `MessagesPlaceholder` 用于在 prompt 模板中占位，接收消息列表（而非纯文本）
- ✅ LCEL 用 `|` 连接多个 `Runnable`，可以在中间插入自定义函数（如 `print_prompt`）做调试或处理
- ✅ `input_messages_key` / `history_messages_key` 要和 prompt 模板里的变量名对应一致
- ⚠️ `InMemoryChatMessageHistory` + 全局字典 = 仅适合本地测试/单进程场景，重启即丢失，不持久化

## 5. 可以进一步思考/动手验证的问题

1. 如果把 `store` 换成写入本地 JSON 文件或 Redis，要怎么改 `get_history` 函数？
2. 如果两个不同的 `session_id` 同时对话，历史会不会互相干扰？（提示：不会，字典按 key 隔离）
3. `print_prompt` 这个调试函数如果去掉，对链路结果有没有影响？（提示：没有，它只是"旁路打印"，原样返回输入）
4. 尝试自己运行代码，观察第三次提问"总共有几只猪"，模型实际给出的答案是什么，并思考它是依据哪一轮历史信息得出的。
