# 27 - 向量检索构建提示词

> 学习如何将向量检索到的内容作为上下文，构建提示词传给大模型，实现 RAG（检索增强生成）的完整流程

---

## 📚 一、代码概述

本代码演示了**RAG（检索增强生成）的完整流程**：用户提问 → 向量库检索相关内容 → 将检索结果作为上下文构建提示词 → 传给大模型生成回答。

这是 RAG 应用最核心的流程，也是向量数据库最常见的用法。

### 1.1 核心功能

- ✅ 使用 `add_texts` 直接添加文本到向量库
- ✅ 相似度搜索获取相关文档
- ✅ 将检索结果拼接成上下文
- ✅ 构建带上下文的提示词
- ✅ 调用大模型生成回答

## 🔧 二、核心代码解析

### 2.1 导入相关类

```python
from langchain_community.chat_models import ChatTongyi
from langchain_core.vectorstores import InMemoryVectorStore
from langchain_community.embeddings import DashScopeEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
```

| 导入类 | 说明 |
|-------|------|
| `ChatTongyi` | 通义千问聊天模型 |
| `InMemoryVectorStore` | 内存向量存储 |
| `DashScopeEmbeddings` | 通义千问 Embedding 模型 |
| `ChatPromptTemplate` | 聊天提示词模板 |
| `StrOutputParser` | 字符串输出解析器 |

### 2.2 初始化模型

```python
model = ChatTongyi(model="qwen-plus")
```

### 2.3 创建提示词模板

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "以我提供的已知参考资料为主，简清和专业的回答用户问题。参考资料：{context}"),
    ("user", "用户提问{input}")
])
```

**模板中有两个变量**：
- `{context}`：向量库检索到的参考资料
- `{input}`：用户的提问

> 💡 这就是 RAG 的核心——把检索到的知识"喂"给大模型，让它基于这些知识回答问题。

### 2.4 创建向量库并添加数据

```python
vector_store = InMemoryVectorStore(
    embedding=DashScopeEmbeddings(model="text-embedding-v4")
)

# 直接添加文本（add_texts）
vector_store.add_texts([
    "减肥就是要少吃多练",
    "在减脂期间吃东西很重要，清淡少油控制卡路里摄入并运动起来",
    "跑步是很好的运动哦"
])
```

**`add_texts` 方法**：
- 直接传入字符串列表
- 自动对每个文本进行向量化
- 自动存储到向量库
- 比 `add_documents` 更简单，不需要先创建 Document 对象

**对比**：

| 方法 | 输入 | 适用场景 |
|-----|------|---------|
| `add_texts` | 字符串列表 | 简单文本，不需要 metadata |
| `add_documents` | Document 对象列表 | 需要 metadata、来源等信息 |

### 2.5 向量检索

```python
input_text = "怎么减肥？"

# 检索向量库
result = vector_store.similarity_search(input_text, 1)
```

**参数**：
- 第一个参数：用户的问题
- 第二个参数（k）：返回最相似的前 k 个结果

**返回值**：`List[Document]`，Document 对象列表

### 2.6 拼接检索结果

```python
reference_text = "["
for doc in result:
    reference_text += doc.page_content
reference_text += "]"
```

把检索到的多个文档内容拼接成一个字符串，作为提示词中的 `{context}` 变量。

> 💡 这里的拼接方式比较简单，实际项目中通常会用更清晰的分隔符，比如空行或编号。

### 2.7 调试函数（打印提示词）

```python
def print_prompt(prompt):
    print(prompt.to_string())
    print("=" * 20)
    return prompt
```

这是一个调试用的函数，用于打印构建好的提示词，方便查看上下文是否正确。

### 2.8 构建链并调用

```python
chain = prompt | print_prompt | model | StrOutputParser()

res = chain.invoke({
    "input": input_text,
    "context": reference_text
})

print(res)
```

**链的组成**：

```
输入字典（input + context）
    ↓
prompt（填充模板，生成消息列表）
    ↓
print_prompt（打印调试）
    ↓
model（调用大模型）
    ↓
StrOutputParser（提取文本内容）
    ↓
输出：回答字符串
```

## 💡 三、关键知识点

### 3.1 什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种让大模型基于外部知识回答问题的技术。

**为什么需要 RAG？**

| 问题 | 说明 |
|-----|------|
| **知识过时** | 大模型的训练数据有截止日期，不知道最新信息 |
| **私有知识** | 大模型不知道你公司的内部文档、产品资料 |
| **幻觉问题** | 大模型可能会"一本正经地胡说八道" |
| **数据安全** | 不能把所有私有数据都喂给大模型训练 |

**RAG 的解决方案**：

```
用户提问
    ↓
[检索] 从知识库中找相关内容
    ↓
[增强] 把相关内容加到提示词里
    ↓
[生成] 大模型基于上下文回答问题
```

> 💡 简单说：先"搜"相关资料，再"读"资料，最后"答"问题。

### 3.2 RAG 的完整流程

```
┌─────────────────────────────────────────────────┐
│                数据准备阶段（离线）                │
├─────────────────────────────────────────────────┤
│  文档 → 分割 → 向量化 → 存入向量库                 │
└─────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────┐
│                问答阶段（在线）                    │
├─────────────────────────────────────────────────┤
│  用户提问 → 向量化 → 相似度搜索 → 取 Top K          │
│                         ↓                        │
│  构建提示词（问题 + 检索到的上下文）                 │
│                         ↓                        │
│  大模型生成回答 → 返回给用户                        │
└─────────────────────────────────────────────────┘
```

### 3.3 提示词构建的要点

**好的 RAG 提示词应该包含**：

1. **角色设定**：告诉大模型它是什么角色
2. **使用规则**：要求它只能基于提供的资料回答
3. **参考资料**：检索到的上下文内容
4. **用户问题**：用户的具体提问

**示例模板**：

```
你是一个专业的知识助手。

请根据以下参考资料回答用户的问题。
要求：
1. 只能使用参考资料中的信息
2. 如果参考资料中没有答案，请回答"根据现有资料无法回答"
3. 回答要简洁准确

参考资料：
{context}

用户问题：{question}

回答：
```

> 💡 明确告诉大模型"不知道就说不知道"，可以有效减少幻觉。

### 3.4 add_texts vs add_documents

| 对比项 | add_texts | add_documents |
|-------|-----------|---------------|
| **输入** | 字符串列表 | Document 对象列表 |
| **metadata** | 没有（或单独传） | 每个 Document 有自己的 metadata |
| **复杂度** | 简单 | 稍复杂 |
| **适用场景** | 简单文本、快速测试 | 正式项目、需要来源追踪 |

**add_texts 示例**：
```python
vector_store.add_texts([
    "文本1",
    "文本2",
    "文本3"
])
```

**add_documents 示例**：
```python
from langchain_core.documents import Document

vector_store.add_documents([
    Document(page_content="文本1", metadata={"source": "doc1"}),
    Document(page_content="文本2", metadata={"source": "doc2"}),
])
```

## 🎯 四、常见使用模式

### 4.1 最简 RAG 实现

```python
from langchain_core.vectorstores import InMemoryVectorStore
from langchain_community.embeddings import DashScopeEmbeddings
from langchain_community.chat_models import ChatTongyi
from langchain_core.prompts import ChatPromptTemplate

# 1. 准备向量库
vector_store = InMemoryVectorStore(embedding=DashScopeEmbeddings())
vector_store.add_texts(["知识1", "知识2", "知识3"])

# 2. 检索
question = "用户的问题"
docs = vector_store.similarity_search(question, k=2)
context = "\n".join([doc.page_content for doc in docs])

# 3. 构建提示词并回答
prompt = f"根据以下资料回答问题：\n{context}\n\n问题：{question}"
model = ChatTongyi(model="qwen-plus")
answer = model.invoke(prompt)
print(answer.content)
```

### 4.2 用 LCEL 构建 RAG 链

```python
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# 创建检索器
retriever = vector_store.as_retriever(search_kwargs={"k": 3})

# 格式化文档
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

# 构建 RAG 链
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | model
    | StrOutputParser()
)

# 提问
answer = rag_chain.invoke("什么是 RAG？")
print(answer)
```

> 💡 这是 LangChain 官方推荐的 RAG 写法，用 `as_retriever()` 把向量库转成检索器，再用 LCEL 组合成链。

### 4.3 带来源引用的回答

```python
# 检索时保留 metadata
docs = vector_store.similarity_search(question, k=3)

# 构建带来源的上下文
context_parts = []
for i, doc in enumerate(docs, 1):
    context_parts.append(f"[{i}] {doc.page_content}")
    context_parts.append(f"来源：{doc.metadata.get('source', '未知')}")
context = "\n\n".join(context_parts)

# 提示词要求引用来源
prompt = f"""
根据以下参考资料回答问题，并在答案末尾标注引用来源。

参考资料：
{context}

问题：{question}
"""
```

```python
# 历史对话 + 当前问题 + 检索上下文
```

## ⚠️ 五、注意事项

1. **检索数量 k 要合适**：太少信息不够，太多会有噪音，一般 3-5 个比较合适
2. **上下文要清晰分隔**：多个检索结果之间要用空行或编号隔开，方便大模型区分
3. **要明确使用规则**：告诉大模型"只能用参考资料回答"，减少幻觉
4. **处理找不到答案的情况**：提示词中要说明"没有答案就说不知道"
5. **检索质量决定回答质量**：检索到的内容不相关，回答肯定不对
6. **注意上下文长度限制**：检索的内容加问题不能超过大模型的上下文窗口

## 📝 六、进阶优化技巧

### 6.1 重排序（Rerank）

先粗筛出较多结果，再用更精确的模型重排序：

```python
# 1. 先检索出 10 个候选
candidates = vector_store.similarity_search(question, k=10)

# 2. 用重排序模型选出最相关的 3 个
# （需要接入重排序服务，如 Cohere Rerank 等）
top_docs = rerank(question, candidates, top_k=3)

# 3. 用这 3 个构建提示词
context = format_docs(top_docs)
```

### 6.2 多查询检索（Multi-Query）

把用户问题改写成多个不同角度的查询，分别检索再合并：

```python
# 让大模型把问题改写成 3 个不同的查询
queries = generate_queries(question)  # ["查询1", "查询2", "查询3"]

# 每个查询都检索
all_docs = []
for q in queries:
    docs = vector_store.similarity_search(q, k=3)
    all_docs.extend(docs)

# 去重后作为最终上下文
unique_docs = deduplicate(all_docs)
```

### 6.3 混合检索（Hybrid Search）

结合关键词检索和向量检索：

```python
# 向量检索
vector_results = vector_store.similarity_search(question, k=5)

# 关键词检索（如 BM25）
keyword_results = bm25_search(question, k=5)

# 合并结果
combined = merge_results(vector_results, keyword_results)
```

### 6.4 上下文压缩（Context Compression）

检索到的文档可能很长，只提取和问题相关的部分：

```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

compressor = LLMChainExtractor.from_llm(model)
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=vector_store.as_retriever()
)

# 检索时会自动压缩，只保留相关片段
compressed_docs = compression_retriever.get_relevant_documents(question)
```

## 🔍 七、完整 RAG 示例

```python
from langchain_community.document_loaders import TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_core.vectorstores import InMemoryVectorStore
from langchain_community.embeddings import DashScopeEmbeddings
from langchain_community.chat_models import ChatTongyi
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# ========== 1. 数据准备 ==========
print("=== 1. 准备数据 ===")

# 加载文档
loader = TextLoader("./data/knowledge.txt", encoding="utf-8")
docs = loader.load()

# 分割文本
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
split_docs = splitter.split_documents(docs)
print(f"分割成 {len(split_docs)} 个文档块")

# 存入向量库
vector_store = InMemoryVectorStore(embedding=DashScopeEmbeddings())
vector_store.add_documents(split_docs)
print("向量库构建完成")

# ========== 2. 构建 RAG 链 ==========
print("\n=== 2. 构建 RAG 链 ===")

# 提示词模板
prompt = ChatPromptTemplate.from_template("""
你是一个专业的知识助手。请根据以下参考资料回答用户的问题。

要求：
1. 只能使用参考资料中的信息
2. 如果参考资料中没有答案，请回答"根据现有资料无法回答这个问题"
3. 回答要简洁明了，重点突出

参考资料：
{context}

用户问题：{question}

回答：
""")

# 模型
model = ChatTongyi(model="qwen-plus")

# 检索器
retriever = vector_store.as_retriever(search_kwargs={"k": 3})

# 格式化文档函数
def format_docs(docs):
    return "\n\n---\n\n".join(doc.page_content for doc in docs)

# 构建 RAG 链
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | model
    | StrOutputParser()
)

print("RAG 链构建完成")

# ========== 3. 问答测试 ==========
print("\n=== 3. 问答测试 ===")

questions = [
    "什么是 RAG？",
    "向量数据库有什么用？",
    "LangChain 是什么？",
    "今天天气怎么样？"  # 测试"不知道"的情况
]

for q in questions:
    print(f"\n问：{q}")
    answer = rag_chain.invoke(q)
    print(f"答：{answer}")
```

## 💡 八、RAG 效果评估

### 8.1 评估维度

| 维度 | 说明 |
|-----|------|
| **检索准确率** | 检索到的内容是否真的相关 |
| **回答准确性** | 回答是否正确，有没有幻觉 |
| **回答完整性** | 是否完整回答了问题 |
| **回答相关性** | 回答是否和问题相关 |
| **引用准确性** | 是否正确引用了资料 |

### 8.2 常见问题与解决

| 问题 | 原因 | 解决方法 |
|-----|------|---------|
| 回答不相关 | 检索到的内容不对 | 优化 Embedding 模型、调整 chunk 大小 |
| 回答不完整 | 检索到的内容不全 | 增加 k 值、优化分割策略 |
| 出现幻觉 | 大模型自己"编"答案 | 加强提示词约束、增加引用检查 |
| 回答太啰嗦 | 提示词没要求简洁 | 在提示词中明确"简洁回答" |
| 找不到答案 | 知识库没有相关内容 | 扩充知识库、明确"不知道"的回复 |
